3. ARQUITECTURA: SERVIDOR LOCAL + CLIENTE
--------------------------------------------------------------------------------

Aunque no haya multijugador activo, el juego se ejecuta de la siguiente forma:

• Se crea una instancia de servidor local (en la misma máquina).
• El jugador se "une" a ese servidor local como si fuera un cliente.
• Toda la lógica de partida (mundo, entidades, jugador, recursos, etc.) vive en el servidor; el cliente muestra y envía inputs.

ESTADO ACTUAL DE IMPLEMENTACIÓN
------------------------------------
- GameServer.cs: Sistema servidor local completo (475 líneas)
  - PlayerInfo: Gestión de jugadores conectados (PlayerId, Client, Stream, Position, Rotation, LastActivity)
  - StartServerWithPortFallback(): Inicio con fallback automático de puertos
  - ProcessServerTick(): Sistema de ticks a 60Hz
  - ProcessBlockedPosition(): Procesamiento de posiciones bloqueadas
  - IDs automáticos: player_1, player_2, etc.
  - CultureInvariant: Parsing numérico robusto
  - Broadcast a todos los jugadores
- GameClient.cs: Sistema cliente con conexión a servidor local (527 líneas)
  - RemotePlayer: Gestión de jugadores remotos (PlayerId, Position, Rotation)
  - Protocolo TCP: Host 127.0.0.1, Puerto 7778
  - Mensajes: STATE_INIT, STATE_UPDATE, PLAYER_STATE, PLAYER_JOINED, PLAYER_LEFT
  - Formato: "INPUT:movimiento:x,y,z|rotacion:pitch,yaw"
  - Señales: OnPositionUpdated, OnRotationUpdated, OnLocalPlayerIdAssigned
  - CultureInvariant: Parsing numérico robusto
- NetworkManager.cs: Gestión centralizada de red (235 líneas)
  - Orquesta comunicación entre cliente y servidor
  - Gestión de modo red/local automático
  - Sincronización de posición y rotación con servidor
  - Señales: NetworkModeChanged, LocalPlayerIdAssigned, ServerPositionUpdated, ServerRotationUpdated
  - ProcessNetworkMovement(): Procesamiento de inputs en modo red
  - NotifyBlockedPosition(): Notificación de posiciones bloqueadas al servidor
  - Integración directa con PlayerController para aplicar cambios de red
- Comunicación red básica entre cliente y servidor establecida
- Gestión automática de conexiones y desconexiones implementada
- Flujo servidor local + cliente local integrado en el juego
- Base para multijugador cooperativo establecida y funcionando

VENTAJAS DE ESTE ENFOQUE
---------------------------
- La lógica de juego está centralizada en el servidor desde el inicio.
- Un posible futuro modo cooperativo o multijugador se reduce a permitir que el servidor escuche en red (o que otros se conecten).
- Mantener la misma lógica de servidor; solo cambia quién se conecta.
- Patrón similar al usado por Minecraft (servidor local + cliente local en singleplayer).

--------------------------------------------------------------------------------
4. FLUJO DE EJECUCIÓN
--------------------------------------------------------------------------------

INICIO DEL JUEGO
- Al ejecutar el juego se carga scenes/main_menu.tscn (definido en run/main_scene).
- Menú principal: "Nueva partida" → GameFlow.OpenNewGameMenu() → cambia a new_game_menu.tscn.
- En nueva partida: semilla (o "Aleatorio"), nombre personaje, género, nombre mundo → "Crear partida" guarda en SessionData y GameFlow.StartNewGame() → game_world.tscn. "Volver" → main_menu.
- "Cargar partida" → por ahora igual que nueva partida (persistencia pendiente).

MENÚS Y NAVEGACIÓN
- "Opciones" → GameFlow.OpenOptions() → options_menu.tscn (panel lateral: Controles, Gráficos; Volver → main_menu).
- "Salir" → GameFlow.QuitGame() → cierra el juego.
- En partida (game_world): tecla Escape (o ui_cancel/pause) → GameFlow.ReturnToMainMenu() → vuelve a main_menu.tscn.

CARACTERÍSTICAS DE LA INTERFAZ
- El juego NO se pausa (continúa ejecutándose) - diseño intencional.
- Sistema CardLayout para transiciones fluidas entre menús.
- Sin cerrar ventanas al cambiar entre menús.
- Botones interactivos con feedback visual.
- Colores consistentes y accesibilidad considerada.
